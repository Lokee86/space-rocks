# Collision Shape Data

Parent index: [Data](./!INDEX.md)

## Purpose

This document describes the collision-shape data source, export workflow, generated handoff file, consumers, validation commands, and ownership boundaries.

## Overview

Collision-shape data bridges Godot-authored scene collision nodes into server-consumed shape data.

The authoring source is the collision geometry stored in Godot scenes. The export tool reads those scenes and writes the shared JSON handoff file:

```text
client scene collision nodes
-> client/tools/export_collision_shapes.gd
-> shared/collisions/collision_shapes.json
-> game-server physics catalog loading
-> runtime collision bodies
```

`shared/collisions/collision_shapes.json` is the persisted generated handoff consumed by the Go game server. It is authoritative for server runtime loading, but it is not the place where collision geometry should normally be edited by hand. Normal edits should be made in the relevant Godot scene collision nodes, then exported again.

This pipeline is separate from `tools/data_sync/`. Collision shapes are not generated from TOML, do not use `tools/data_sync/config.toml`, and do not generate Go or GDScript source files.

## Source files

Authored collision geometry currently lives in these Godot scenes:

```text
client/scenes/bullet.tscn
client/scenes/player.tscn
client/scenes/asteroid.tscn
client/scenes/pickups/powerup_pickup.tscn
client/scenes/pickups/weapon_pickup.tscn
```

The export tool is:

```text
client/tools/export_collision_shapes.gd
```

The exporter reads:

```text
bullet   -> client/scenes/bullet.tscn -> CollisionShape2D
ship     -> client/scenes/player.tscn -> CollisionPolygon2D
asteroid -> client/scenes/asteroid.tscn -> CollisionVariants children
pickups  -> client/scenes/pickups/*_pickup.tscn -> CollisionShape2D
```

Pickup shapes are exported by pickup class:

```text
powerup
weapon
```

Pickup shape keys are class-level keys, not pickup type keys. For example, `powerup` is the collision-shape key, while `1_up` is a pickup type.

## Configuration

Collision export has no external config file.

The exporter currently hardcodes:

```text
OUTPUT_PATH := res://../shared/collisions/collision_shapes.json
BULLET_SCENE := res://scenes/bullet.tscn
PLAYER_SCENE := res://scenes/player.tscn
ASTEROID_SCENE := res://scenes/asteroid.tscn
PICKUP_POWERUP_SCENE := res://scenes/pickups/powerup_pickup.tscn
PICKUP_WEAPON_SCENE := res://scenes/pickups/weapon_pickup.tscn
```

`tools/data_sync/config.toml` does not own collision-shape export.

## Generated outputs

The collision-shape pipeline writes one persisted generated output:

```text
shared/collisions/collision_shapes.json
```

No Go source files are generated by this pipeline.

No GDScript source files are generated by this pipeline.

The game-server files that load and test collision shapes are hand-authored implementation and test files, not generated outputs.

## Data shape

The shared JSON catalog currently has these top-level fields:

```text
bullet
ship
asteroids
pickups
```

The game server imports the catalog into:

```go
type CollisionShapeCatalog struct {
    Bullet    ImportedCollisionShape            `json:"bullet"`
    Ship      ImportedCollisionShape            `json:"ship"`
    Asteroids []ImportedCollisionShape          `json:"asteroids"`
    Pickups   map[string]ImportedCollisionShape `json:"pickups"`
}
```

Imported collision shapes use:

```go
type ImportedCollisionShape struct {
    Name   string      `json:"name"`
    Type   string      `json:"type"`
    Radius float64     `json:"radius"`
    Height float64     `json:"height"`
    Size   []float64   `json:"size"`
    Points [][]float64 `json:"points"`
}
```

Supported imported shape types are:

```text
circle
capsule
rectangle
polygon
```

Current exported shape usage:

```text
bullet   -> capsule
ship     -> polygon
asteroid -> polygon list
pickups  -> circle shapes keyed by pickup class
```

The pickup JSON entries currently include `offset`, but the Go import struct does not consume it. Server pickup collision bodies use the runtime pickup position plus the imported circle geometry.

## Consumers

The game server loads the shared JSON through:

```text
services/game-server/internal/game/physics/collision_shapes.go
```

`physics.LoadCollisionShapeCatalog()` searches upward from the process working directory until it finds:

```text
shared/collisions/collision_shapes.json
```

`game.New()` loads the catalog once for the game aggregate and stores it as:

```text
Game.collisionShapes
```

Runtime entity collision-body builders consume the loaded catalog:

```text
runtime.Ship.CollisionBody(...)
runtime.Bullet.CollisionBody(...)
runtime.Asteroid.CollisionBody(...)
pickups.Pickup.CollisionBody(...)
```

Gameplay systems then consume those bodies for:

```text
projectile -> asteroid collision checks
player -> asteroid collision checks
player -> pickup collection checks
server-side target click validation
safe initial spawn and respawn placement
radial candidate radius derivation
devtools collision telemetry
```

Devtools shape catalog output also reloads the shared catalog when building the debug shape catalog response.

The client does not consume `shared/collisions/collision_shapes.json` as runtime source data. The client owns the scene nodes that author and present collision geometry.

## Pipeline usage

When collision geometry changes:

1. Edit the relevant Godot scene collision node.
2. Run the collision export tool from the repo root:

```bash
godot --headless --path client -s res://tools/export_collision_shapes.gd
```

3. Review the diff for:

```text
shared/collisions/collision_shapes.json
```

4. Run focused server tests that cover collision-shape loading and consumers.
5. Run client tests if the scene edit also changes client presentation behavior.

Do not hand-edit `shared/collisions/collision_shapes.json` as the normal workflow. A hand edit should be treated as an intentional generated-output patch and should be replaced by an exporter/source-scene fix when possible.

## Validation commands

Export collision shapes:

```bash
godot --headless --path client -s res://tools/export_collision_shapes.gd
```

Focused game-server physics validation:

```bash
cd services/game-server
go test -buildvcs=false ./internal/game/physics ./tests/physics
```

Focused gameplay consumer validation:

```bash
cd services/game-server
go test -buildvcs=false ./internal/game ./tests/game -run 'Collision|Pickup|Respawn|Target|ShipCollisionShape'
```

Full game-server validation:

```bash
cd services/game-server
go test -buildvcs=false ./...
```

`data-sync -check` does not validate collision-shape export. Collision shapes are outside the data-sync managed TOML pipeline.

## Failure modes

Common export failure modes include:

```text
missing source scene
missing expected CollisionShape2D node
missing expected CollisionPolygon2D node
missing asteroid CollisionVariants node
unsupported Godot shape type
failed write to shared/collisions/collision_shapes.json
```

Common shared JSON failure modes include:

```text
stale JSON after scene collision edits
invalid JSON
missing bullet shape
missing ship shape
empty asteroid shape list
empty pickup shape catalog
pickup shape keyed by pickup type instead of pickup class
unsupported imported shape type
invalid circle radius
invalid capsule radius or height
invalid rectangle size
invalid polygon point shape
```

Runtime failure behavior is intentionally non-fatal at game construction time. If `Game.New()` cannot load the catalog, it logs a warning and creates the game with an empty catalog. Collision-dependent body construction can then fail, and gameplay consumers skip unavailable bodies instead of inventing fallback geometry at the call site.

## Notes

The legacy source-of-truth map correctly classified collision shapes as active shared collision data, generated by the Godot export tool and consumed by server collision loading.

The durable ownership split is that Godot scenes author the geometry, the export tool writes the JSON handoff, the game server imports the handoff into runtime collision primitives, and gameplay systems decide what collision results mean.

Collision-shape data does not own gameplay rules, packet shapes, database schemas, damage resolution, scoring, pickup effects, or client presentation behavior.
