# Data Sync And SSoT Pipeline

Parent index: [Data](./!INDEX.md)

## Purpose

This document describes the current Space Rocks source-of-truth and data-sync pipeline.

It covers the project-wide data ownership boundaries, the active `tools/data_sync` generation workflow, generated outputs, validation commands, and the adjacent source-of-truth paths that are intentionally not owned by data-sync.

## Overview

Space Rocks uses editable shared source files to keep client, server, player-data, and API contracts aligned.

The active `tools/data_sync` pipeline currently owns generation and drift checks for:

```text
constants
packets
drop_tables
```

The data-sync tool reads TOML sources under `shared/`, renders deterministic language outputs, and either writes them, diffs them, or verifies that generated outputs are current.

The broader project also has source-of-truth files that sit beside data-sync but are not generated by it. Those include OpenAPI HTTP contracts, collision-shape exports, asteroid variant data, Rails migrations, Godot scenes, and player-data physical storage schemas.

The important rule is:

```text
Edit the owning source-of-truth file.
Regenerate or validate the derived outputs.
Do not hand-edit generated files as authority.
Do not treat one source-of-truth domain as owning another.
```

## Configuration

The default data-sync configuration is:

```text
tools/data_sync/config.toml
```

That configuration defines:

```text
[sot.constants]
[sot.packets]
[sot.drop_tables]
[sot.player_data]
[constants.scan]
[packets.go]
[packets.gds]
[packets.ts]
[drop_tables.go]
```

`[sot.*]` sections define source-of-truth input paths.

`[constants.scan]` defines where constants destination blocks are discovered. Current scan targets are:

```text
services/game-server/internal/constants/*.go
client/scripts/generated/constants/*.gd
```

Packet target configuration defines generated packet output groups. The packet output IDs are resolved against `shared/packets/outputs.toml`.

Current packet output IDs are:

```text
server_entities_packets
server_game_packets
server_devtools_packets
player_data_packets
client_packets
```

TypeScript packet output is configured but disabled:

```text
[packets.ts]
enabled = false
```

Drop tables currently generate Go output only.

## Pipeline usage

Exactly one data-sync operation must be selected:

```text
-push
-pull
-diff
-check
-validate
```

Supported active generation domains are:

```text
-constants
-packets
-drop-tables
```

`-player_data` is currently validation-only.

Supported language flags are:

```text
-go
-gds
-ts
```

`-push` writes generated output.

```text
data-sync -push -constants -go -gds
data-sync -push -packets -go -gds
data-sync -push -drop-tables -go
```

`-diff` renders generated output and prints a unified diff without writing.

```text
data-sync -diff -constants -go -gds
data-sync -diff -packets -go -gds
data-sync -diff -drop-tables -go
```

`-check` renders generated output, writes nothing, exits `0` when outputs are current, and exits `1` when generated files differ.

```text
data-sync -check -constants -go -gds
data-sync -check -packets -go -gds
data-sync -check -drop-tables -go
```

`-validate` checks configuration, source TOML structure, supported values, configured files, managed blocks, packet schemas, drop-table schemas, and player-data logical schema where requested.

```text
data-sync -validate
data-sync -validate -constants
data-sync -validate -packets
data-sync -validate -drop-tables
data-sync -validate -player_data
```

`-pull` is restricted to constants and one language at a time.

```text
data-sync -pull -constants -go
data-sync -pull -constants -gds
```

Packet pull is intentionally unsupported. Packet shape changes must be made in `shared/packets/*.toml` and pushed outward.

Drop-table pull is not the active workflow. Drop-table changes should be made in `shared/drop_tables/*.toml`.

Player-data schema push/pull/diff/check is not implemented. `-player_data` is validation-only.

## Validation commands

Project-wide data-sync validation:

```text
data-sync -validate
```

Constants drift check:

```text
data-sync -check -constants -go -gds
```

Packet drift check:

```text
data-sync -check -packets -go -gds
```

Drop-table drift check:

```text
data-sync -check -drop-tables -go
```

Player-data logical schema validation:

```text
data-sync -validate -player_data
```

Tool test suite:

```text
cd tools/data_sync && python -m pytest
```

Game-server validation after generated server constants, packets, drop tables, asteroid data, or collision data change:

```text
cd services/game-server && go test -buildvcs=false ./...
```

Player-data validation after player-data packets or logical player-data schema change:

```text
cd services/player-data && go test ./...
```

HTTP contract validation after OpenAPI changes:

```text
cd services/api-server && bundle exec rails test test/contracts/openapi_contract_test.rb
```

Collision-shape export validation uses the Godot export tool, not data-sync:

```text
godot --headless --path client -s res://tools/export_collision_shapes.gd
```

## Failure modes

Duplicate constants sections across multiple source TOML files fail validation or generation. Each constants section must have exactly one source owner.

Missing constants sections fail when a generated destination block references a section that does not exist in the constants source set.

Malformed or ambiguous managed blocks fail constants sync. Missing start markers, missing end markers, duplicate blocks, or mismatched marker comment styles are hard failures.

Unsupported constants values fail validation. Constants currently support booleans, integers, floats, strings, and two-number vector-style arrays.

Packet schema errors fail validation or generation. Common failures include duplicate packet type IDs, duplicate packet values, invalid struct names, invalid JSON field names, unsupported field types, unknown struct references, unknown packet output IDs, or references to unknown packet builder fields.

Packet target errors fail generation. A configured packet target must be enabled, must have a matching output in `shared/packets/outputs.toml`, and must use a supported language renderer.

Packet pull fails by design. Packet schemas are edited in TOML, not reconstructed from generated Go or GDScript.

Drop-table generation is Go-only. `-drop-tables` requires `-go`.

Player-data schema currently validates only a limited logical schema shape. Supported player-data field types are:

```text
string
integer
boolean
```

Each player-data field must define `required`, `optional`, or `default`.

TypeScript packet output is disabled in the default configuration. A command that tries to generate disabled packet targets should fail instead of silently producing partial output.

Editing generated files directly creates drift. The correct fix is to update the owning source and rerun the appropriate data-sync command.

Treating OpenAPI, Rails migrations, collision exports, or Godot scene data as data-sync-owned material creates ownership confusion. Those are adjacent source-of-truth paths with separate validation workflows.

## Notes

Legacy data-sync docs layout links should now point at this document for workflow guidance, or at `tools/data_sync/config.toml` when the configuration file itself is the subject.

`SSoT` in this document means source of truth. It does not imply every source of truth is handled by `tools/data_sync`.

The data-sync pipeline is intentionally narrower than the project-wide source-of-truth map. Constants, packets, and drop tables are active generated domains. Player-data logical schema is validate-only. HTTP contracts, collision exports, asteroid variants, Rails migrations, and Godot scenes use separate ownership and verification paths.
