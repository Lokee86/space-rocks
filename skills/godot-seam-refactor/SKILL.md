# Godot Seam Refactor Skill

Use this skill when extracting, shrinking, routing, or reorganizing Godot client scripts, especially large gameplay, UI, lifecycle, networking, sync, shell, or game-loop files.

## When to use

Use this skill for work involving files such as:

- `client/scripts/gameplay/game.gd`
- `client/scripts/game.gd`
- `client/scripts/ui/game_shell.gd`
- `client/scripts/ui/hud_controller.gd`
- `client/scripts/networking/world_sync.gd`
- `client/scripts/networking/network_client.gd`
- spectate, lifecycle, sync, HUD, menu, presentation, and controller scripts

## Core rules

- Preserve behavior first. Extract or route existing behavior before adding new behavior.
- Prefer pure/helper/controller extraction before scene ownership changes.
- Avoid risky scene/node/path/signal changes unless required.
- Folder moves are less important than scene wiring, node paths, signal connections, script attachments, preload paths, and lifecycle timing.
- Do not create vague buckets such as `utils`, `helpers`, `manager`, `common`, or `misc`.
- Use concrete ownership names such as `spectate_targets`, `player_lifecycle`, `gameplay_lifecycle_controller`, `packet_codec`, `bullet_sync`, `asteroid_sync`, `player_sync`, or `local_visual_sync`.
- If no obvious owner exists, stop and report the missing seam instead of placing code in the nearest working file.
- Do not let integration/controller seams become god objects.

## Line-count guardrails

For hand-written Godot scripts:

- Prefer under about 200 lines.
- 300+ lines requires responsibility scrutiny.
- 350+ lines should not receive a new responsibility unless the prompt explicitly allows it.
- 500+ lines is a gravity-well warning. Prefer shrinking, routing, or extracting.
- 800+ lines is active god-object risk unless the file is being split.

These limits do not apply to `.tscn`, `.tres`, generated files, addons, fixtures, or snapshots.

## Safe extraction pattern

1. Identify the exact responsibility being extracted.
2. Create or use a script with one concrete owner.
3. Move the smallest behavior-preserving slice.
4. Update `preload(...)` paths and references.
5. Keep public methods narrow and named by behavior, not implementation.
6. Avoid scene rewiring unless the prompt specifically requires it.
7. Update focused GUT tests or add a small pure/unit test when cheap.
8. Run focused GUT tests if the `godot` CLI is available.

## Godot-specific checks

After path or script moves, check for:

```bash
rg "res://old/path" client
rg "OldClassName|old_script_name" client
```

For script extractions, inspect:

- `preload(...)` paths
- `extends` / `class_name`
- signal connections
- node paths and `%UniqueName` references
- `_ready`, `_process`, `_physics_process`, and lifecycle timing
- `.uid` files generated by Godot
- unintended `.tscn` rewrites

## Verification

Prefer focused GUT when available:

```bash
godot --headless --path client -s res://addons/gut/gut_cmdln.gd -gdir=res://tests/unit -ginclude_subdirs -gexit
```

If GUT is unavailable, report that clearly and run focused `rg` checks that prove paths/references were updated.
